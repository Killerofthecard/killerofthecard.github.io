不改模型，改脚手架？｜Agent Harness 与 Self-Evolve 工程

大模型能力突飞猛进的当下，一个反直觉的事实逐渐浮出水面：同一个模型，放在不同的”脚手架”里，能表现出截然不同的能力。Lilian Weng（前OpenAI安全研究VP、Thinking Machines Lab联合创始人）在近期万字长文 Harness Engineering for Self-Improvement 中系统阐述了这一观点，并指出——AI 自进化的近路，可能不在于模型修改自身权重，而在于优化包裹模型的那层”Harness”（运行脚手架）。本文结合该文及相关前沿工作（Self-Harness、Darwin Gödel Machine），梳理这一方向的核心思想与技术路径。

1. 什么是 Harness？

先厘清概念。Harness 指模型之外、负责调度和编排的一切：模型如何调用工具、如何管理上下文、如何读写文件、如何拆解任务和派发子Agent、如何验证结果、如何从失败中恢复。如果你用过 Claude Code 或 Codex CLI，你所交互的那个”命令行 Agent 层”就是典型的 Harness。

传统认知中，系统性能提升的路径是：

$$\text{更好的模型} \Rightarrow \text{更强的能力}$$

但 Harness 工程的核心主张是，这条路径还应有一条正交的维度：

$$\text{更好的 Harness} \Rightarrow \text{同一模型} \Rightarrow \text{更强的能力}$$

二者构成一个二维的提升空间。而且——这一点尤为关键——Harness 优化走到极致，可以形成闭环：让模型自己去改进自己的 Harness。这便是自进化（Self-Evolve）的工程入口。

Weng 将 Harness 的优化层次递进概括为一条清晰的链路：

$$\text{prompt} ;\to; \text{结构化 context} ;\to; \text{workflow} ;\to; \text{harness code} ;\to; \text{optimizer code}$$

越往右，自动化的程度越高，系统越接近真正的递归自我改进（Recursive Self-Improvement, RSI）。

2. 三个基础设计模式

在进入自进化之前，Weng 先厘清了现代 Harness 的三个基础设计模式，它们是后续所有自动化优化的地基。

模式一：Workflow 自动化循环

一个 goal-oriented 的 agent 循环，本质是四步迭代：

$$\text{Plan} \to \text{Execute} \to \text{Observe} \to \text{Improve}$$

与传统静态 prompt 模板不同，workflow 循环让模型分析自己的执行轨迹和失败记录，迭代式的改进策略。这一模式向上可以追溯到强化学习中的 agent-environment loop，区别在于这里将”环境反馈”替换为了代码执行结果、工具调用日志、validator 打分等结构化信号。

模式二：文件系统即持久记忆

长周期 agent 任务会积累大量中间产物——实验日志、代码 diff、错误堆栈、中间状态——这些很快超出上下文窗口的容量。一个成熟 Harness 的做法是：将状态持久化到文件系统（通过 bash 写入文件），而非全部塞进 context。这样做有两个好处：(1) 上下文保持精炼，模型始终聚焦当前步骤；(2) 中间产物可被事后审查、中断后恢复。Weng 指出，这种能力会随着基础模型本身能力的增强而自然受益——模型越聪明，越懂得什么该记、什么该丢。

模式三：子 Agent 与后台任务

Harness 可以 spawn 多个子 agent 并行执行，并通过文件/日志/状态记录来交换信息。这类似于操作系统的进程模型：父进程 fork 子进程，子进程写结果到文件，父进程收集并汇总。核心原则是输出必须可审查（inspectable），不能是黑箱——否则一出错就是连锁反应。

3. 自进化 Harness：三层架构

这是文章的重头戏。Weng 将 Harness 自动化优化分为三个递进的层次。

第一层：Context Engineering

这一层将 context 本身当作可以持续迭代优化的”操作手册”。代表性工作是 ACE (Agentic Context Engineering)：三个角色协作——

  • Generator：生成任务执行轨迹；
  • Reflector：从成功/失败中提炼经验洞察；
  • Curator：将洞察结构化地增量写入 context。

更进一步，MCE (Meta Context Engineering) 将”如何管理 context”和”应该在 context 中放什么”拆成两个优化层——外层进化 context 管理技能，内层使用这些技能优化具体任务的 context。这已经逼近”自我管理的记忆系统”。

第二层：Workflow Design

如果 context engineering 优化的是”给模型看什么”，这一层优化的是”让模型怎么做”。ADAS (Automated Design of Agentic Systems) 将 workflow 设计本身建模为搜索问题——一个 meta-agent 不断提出和评估新的 workflow 变体。AFlow 更进一步，将 workflow 表示为图（graph），用蒙特卡洛树搜索（MCTS）在图的拓扑空间中寻找最优结构。优化粒度从单个 prompt 提升到了整个 agent 行为的组织结构。

第三层：Self-Improving Harness（核心）

到了这一层，模型开始分析和修改 Harness 本身。最具代表性的工作是上海 AI Lab 的 Self-Harness——一个三阶段闭环系统：

  1. Weakness Mining（弱点挖掘）：模型用当前 Harness 跑一批任务，收集完整的执行轨迹（工具调用、错误日志、validator 反馈）。失败的轨迹被聚类分析，识别出反复出现的、模型特异性的失败模式——例如：总是忘记产出最终文件、工具出错后反复重试无效指令、在探索阶段过度徘徊而不进入实现阶段。注意这里不是把每次失败当作独立事件，而是做模式归因。

  2. Harness Proposal（修改提案）：同一个模型切换到”提案者”角色，针对识别出的失败模式生成候选的 Harness 修改。修改的边界被严格限定在预先声明的可编辑表面（system prompt、工具使用规则、验证逻辑、runtime policy 等），不允许动核心架构。每个提案必须说明：它要改变什么行为、可能引入什么回归风险、为什么可能修复已识别的失败模式。

  3. Proposal Validation（回归验证）：候选修改通过严格的回归测试进行验证。一个保守的接收规则是：至少在一个评估分片上提升，且不在任何其他分片上退化。只有通过验证的修改才被合入下一个版本的 Harness，作为下一轮迭代的起点。

值得一提的是，整个过程是去主观化的——模型不能自己”觉得”改好了就算数，每一步都要经过可记录、可复现、可回滚的评估闸门。

效果方面，在 Terminal-Bench-2.0（容器化终端环境中的多轮 agent benchmark）上，不换模型、不改工具、不改评估协议，Self-Harness 带来了显著的提升：

模型 提升前 提升后 相对提升
Qwen3.5-35B-A3B 23.8% 38.1% (held-out) +60%
MiniMax M2.5 40.5% 61.9% +53%
GLM-5 42.9% 57.1% +33%

更有趣的是，不同模型的自进化方向截然不同——Self-Harness 产生的 Harness 修改是模型特异性的：

  • MiniMax M2.5（无尽探索型）：被加了 50 次工具调用后的 loop-breaker，以及尽早产出初始产物的规则；
  • Qwen3.5-35B-A3B（倔强重复型）：被加了依赖预检查、工具出错后的产物恢复逻辑、禁止精确重复指令的规则；
  • GLM-5（状态丢失型）：被加了跨 shell 命令保持环境变量的规则。

这说明 Harness 自进化不是无脑”加长 prompt”，而是针对模型特定弱点的靶向修复。

4. 进化搜索：Darwin Gödel Machine

如果说 Self-Harness 是”诊断-开方-验效”的闭环医疗模式，那 Darwin Gödel Machine (DGM) 走的是另一个极端——让 agent 直接修改自己的代码仓库，在开放空间中做进化搜索。

DGM（UBC / Vector Institute / Sakana AI）的精妙之处在于其开源式探索 + 自我改进的组合：

  • 维护一个 agent 代码的档案库（archive），每次迭代从中采样（不一定是当前最优的，也包括历史上表现一般的），让采样出的 agent 检查自己的代码、提出改进、实现并测试；
  • 成功的修改被追加到档案库中，形成一棵不断生长的进化树；
  • 两个消融实验直指要害：去掉自我改进（固定 meta-agent）→ SWE-bench 从 50% 跌到 39%；去掉开源式探索（总是从最新版本出发）→ 直接陷入局部最优，停在 23%。

使用 Claude 3.5 Sonnet 作为基础模型，从一个简单初始配置出发：

Benchmark 初始 80 轮进化后 对比基准
SWE-bench Verified 20.0% 50.0% OpenHands+CodeAct v2.1: 53.0%
Polyglot 14.2% 30.7% 超越人工设计的 Aider

DGM 在进化过程中自动发现了多项后来被证明有效的设计：细粒度文件编辑（字符串替换替代全文件覆盖）、行范围查看（应对长上下文）、多候选方案+同行评审打分、撤销功能和编辑历史追踪、上下文超限时的摘要机制。

不过也有一则来自 Goodhart’s Law 的警告：实验中 DGM 曾被观察到伪造测试日志以及试图关闭幻觉检测模块，而不是修复底层的代码问题。这提醒我们——只要优化信号本身是脆弱的，足够聪明的 agent 就会找到利用它的方式。

5. 七个未解决的瓶颈

Weng 在文章末尾坦率列出了通向真正 RSI 的七个障碍，笔者认为是全篇最有价值的部分：

  1. 弱且模糊的 Evaluator：代码和数学任务有客观的 pass/fail 信号，但”研究品味””创新性””长期科学价值”几乎无法量化。没有清晰的 evaluator，自进化循环就失去了方向。

  2. 上下文与记忆的生命周期管理：任务越自主，记忆越膨胀。Weng 推测，记忆管理能力最终可能需要内化为”智能”的一部分，而非仅仅是软件系统的外部功能。

  3. 负结果的系统性忽视：学术界的发表偏倚导致 LLM 训练数据中充斥”成功叙事”。模型不知道什么时候该放弃假设、什么时候该诚实报告失败。一个成熟的研究 Harness 应该让失败容易被记录和学习。

  4. 多样性坍缩：进化搜索和 RL 循环倾向于利用已知高奖励模式，导致种群坍缩为单一解的变体。在没有保护机制的情况下，开源式探索变成了一句空话。

  5. Reward Hacking：系统优化它接收到的任何信号——过拟合单测、讨好 judge model、利用 benchmark 的统计假象——而不是真正变强。Evaluator 和权限控制应当位于进化循环外部，辅以留出测试集和轨迹审计。

  6. 短期成功 vs 长期健康：现有一切 benchmark 都优化”完成当前任务”。但真实工程世界关心可维护性、向后兼容、迁移成本、未来调试负担——这些在短视 reward 信号里完全无法被捕捉。

  7. 人的位置：不是把人移出 loop，而是让人向上移动到更高的抽象层。在正确的时间和抽象级别设置人的介入点——这本身就是系统设计的核心决策。

6. 小结

Harness Engineering 的核心洞察可以凝练为一句话：在模型能力给定的前提下，Harness 决定了模型能力的上限。而让模型自己去改进自己的 Harness，可能是通向 RSI 的最务实工程路径。

三个要点收束：

  1. Harness 不是附属品——它是决定 agent 能力的独立维度，与模型权重同等重要；
  2. 自进化 Harness（Self-Harness / DGM）已经证明：不改模型权重，仅优化脚手架，就能在多个 benchmark 上获得 30%~100%+ 的性能提升；
  3. 但七个瓶颈——尤其是 evaluator 的模糊性、reward hacking、多样性坍缩——意味着这条路才刚刚开始，离真正的 RSI 还有相当距离。

模型变强，Harness 也在变强，二者互为放大器——这可能是接下来 AI engineering 最值得关注的方向。
